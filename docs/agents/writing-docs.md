# writing-docs.md

이 저장소의 문서는 "운영 문서"가 아니라 "구현 문서"로 작성한다.  
대상 독자는 이 보일러플레이트를 읽고 기능을 추가하거나 구조를 바꿀 구현자다.

## 기준 문서

문서 톤과 형식은 아래 두 파일을 기준으로 맞춘다.

- `frontend/docs/bff-proxy.md`
- `frontend/docs/auth.md`

이 두 문서는 완성본이라기보다 기준 샘플이다.  
다른 문서를 쓸 때도 같은 결로 유지한다.

## 문서 목적

- 현재 구현이 어떻게 되어 있는지 설명한다.
- 기술을 왜 쓰는지 설명하되, 이 저장소 안에서의 쓰임만 적는다.
- 구현자가 바로 따라갈 수 있게 경로, 인터페이스, 데이터 shape를 보여준다.

다음 종류의 문서는 만들지 않는다.

- 운영 체크리스트
- 장애 대응 문서
- 정책 문서
- 리스크 정리
- 일반론 위주의 기술 소개

## 먼저 확인할 것

문서를 쓰기 전에 반드시 아래 두 가지를 먼저 확인한다.

1. 관련 코드 파일
2. 관련 공식 웹 문서

코드를 읽지 않고 추측으로 문장을 만들지 않는다.  
공식 문서를 확인하지 않고 기억에 의존해 링크를 넣지 않는다.

## 문체

- 문서가 자기 자신을 설명하는 말투를 쓰지 않는다.
- 구현을 읽는 사람의 메모처럼 쓴다.
- 짧은 문장으로 끊는다.
- 추상어보다 실제 경로와 실제 이름을 쓴다.

쓰지 말아야 할 표현:

- `이 문서의 범위`
- `역할`
- `개요`
- `소개`
- `배경`
- `핸들러 모양은 아래와 같다`
- `현재 설정은 아래 모양이다`
- `여기서 중요한 것은`
- `정리하면`
- `즉`
- `다시 말해`
- `빠른 읽기 순서`
- `먼저 볼 것`
- `온보딩`

예시:

- 나쁨: `이 문서의 범위는 BFF 프록시 구현이다.`
- 좋음: ``src/app/api/proxy/[...path]/route.ts`는 브라우저 쿠키 요청을 JWT 요청으로 바꾼다.`

## 구조

설명만 길게 쓰지 않는다.  
중간중간 실제 구조가 보이는 코드 블록을 넣는다.

반드시 넣을 수 있는 것들:

- route handler 시그니처
- config object
- DTO shape
- env schema
- 요청 흐름
- 경로 변환
- 응답 예시

예시:

```ts
export const config = {
  matcher: ["/dashboard/:path*"],
};
```

```text
/api/proxy/post/api/posts
-> http://localhost:8080/api/post/api/posts
-> nginx location /api/post/
-> post-service:8080
```

## 섹션 제목

섹션 제목도 구현 이름으로 바로 들어간다.

좋은 제목:

- `## /api/auth/[...all]`
- `## JWT`
- `## 헤더`
- `## body`
- `## Drizzle 테이블`
- `## gateway 경로`

피해야 할 제목:

- `## 범위`
- `## 역할`
- `## 정책`
- `## 체크리스트`
- `## 운영 시 확인 사항`
- `## 한계`
- `## 범위 밖`

## 파일 경로와 링크

`주요 파일` 섹션에는 상대경로만 적는다.

예시:

- `src/app/api/proxy/[...path]/route.ts`
- `src/features/auth/lib/auth.ts`

`참고 문서` 섹션에는 웹 링크만 적는다.  
링크는 공식 문서 위주로 넣는다.

예시:

- Better Auth Generic OAuth: `https://better-auth.com/docs/plugins/generic-oauth`
- Next.js Route Handlers: `https://nextjs.org/docs/app/getting-started/route-handlers`

## 내용 선택 기준

문서에 넣는 내용:

- 현재 코드에 실제로 있는 것
- 실제 경로, 실제 설정값, 실제 타입, 실제 호출 흐름
- 이 서비스 안에서 기술이 맡는 역할

문서에 넣지 않는 내용:

- 구현에 없는 기능
- 미래 계획
- 추측
- 프레임워크 일반론
- "보통은 이렇게 한다" 같은 문장

## 최종 점검

문서를 저장하기 전에 아래를 확인한다.

- 메타 설명 문장이 들어가 있지 않은가
- 코드 블록 없이 말로만 설명한 부분이 남아 있지 않은가
- 실제 파일명, 경로, 엔드포인트 이름이 빠져 있지 않은가
- `주요 파일`은 상대경로만 썼는가
- `참고 문서`는 웹 링크만 썼는가
- 운영/정책/체크리스트 성격의 섹션이 들어가 있지 않은가
