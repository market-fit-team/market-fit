# git-conventions.md

## 커밋 규칙

커밋 전 반드시 스테이징된 모든 변경사항을 확인한 후 이를 대표할 수 있는 커밋명과 커밋 내용으로 작성하여야 한다. 단 여러 타입이 겹친 변경사항의 경우 변경범위가 크고 대표적인 변경사항을 중심으로 작성하고 작은 변경사항은 내용으로 기재할 수 있다.

커밋을 작성할 때에는 아래 규칙의 멀티라인 커밋 형식을 사용한다.

```text
타입: 한국어 요약

- 한국어 상세 내용
- 한국어 상세 내용
```

타입 종류: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `ui`

- `ui` - 비즈니스 로직 변경 없이 디자인/스타일만 변경
- `feat` - 사용자에게 영향을 주는 새로운 기능 추가
- `chore` - 빌드 설정, 의존성 패키지 관리, 환경 설정 등 관리성 작업 변경
- `test` - 테스트 코드 추가 및 수정
- `refactor` - 테스트 코드 및 기능(사용자 경험) 변경 없는 코드 구조 개선
- `fix` - 버그 및 장애 원인 수정
- `docs` - 문서 추가 및 수정

한국어 상세 내용은 생략될 수 있으며, 기재하는 경우 개조식 문체(명사형 종결어미 등)으로 기재.

## 리베이스 및 푸시 전략

작업 브랜치를 원격 저장소에 푸시(Push)하기 전에 메인 브랜치(`origin/main`)를 기준으로 리베이스를 실행하여 선형 히스토리를 유지한다.

### 리베이스 프로세스

```bash
git fetch origin
git rebase origin/main
```

### 충돌 해결 규칙

리베이스 중 충돌이 발생하면 다음 방식으로 해결한다.

1. **사용자 동의**: 충돌 지점을 마음대로 해결하지 않는다. 사용자에게 상황을 알리고 명시적인 동의를 얻어 수정을 진행한다.
2. **main 브랜치 코드 존중**: `main` 브랜치의 변경 사항을 최대한 존중하며, 기존 코드를 파괴하지 않고 안전하게 병합한다.
3. **리베이스 재개**: 충돌이 해결된 후 스테이징 및 리베이스를 계속 진행한다.

```bash
git add .
git rebase --continue
```

### 원격 푸시

- 강제 푸시(Force Push)는 최대한 지양한다.
- 자신의 기능 브랜치에 한해서만 `--force-with-lease` 옵션을 사용하여 푸시할 수 있다.
- `main` 브랜치에는 강제 푸시를 절대로 금지한다.

```bash
git push origin <브랜치명> --force-with-lease
```
